## 개발 시작 전 필수 설정 (초기 세팅)

레포지토리를 처음 복사하거나 원격에서 당겨온 후, 노드 모듈 폴더가 없어서 실행이 안 되는 것을 방지하기 위해 아래 명령어들을 터미널에 순서대로 입력해 주세요.

### 1. 기본 모듈 설치
- 명령어: npm install

### 2. 리액트 라우터 설치 (페이지 이동 및 링크 연동용)
- 명령어: npm install react-router-dom

## 협업 규칙 (Git Convention)

### 1. 커밋 메시지 타입 (Commit Message Type)
우리는 코드의 변경 사항을 명확히 알기 위해 아래 표에 맞춰 커밋 메시지 첫머리를 작성합니다.

| 타입 이름 | 내용 |
| :--- | :--- |
| **feat** | 새로운 기능에 대한 커밋 |
| **fix** | 버그 수정에 대한 커밋 |
| **build** | 빌드 관련 파일 수정 / 모듈 설치 또는 삭제에 대한 커밋 |
| **chore** | 그 외 자잘한 수정에 대한 커밋 |
| **ci** | ci 관련 설정 수정에 대한 커밋 |
| **docs** | 문서 수정에 대한 커밋 |
| **style** | 코드 스타일 혹은 포맷 등에 관한 커밋 |
| **refactor** | 코드 리팩토링에 대한 커밋 |
| **test** | 테스트 코드 수정에 대한 커밋 |
| **perf** | 성능 개선에 대한 커밋 |

> **작성 예시:** `feat: 로그인 기능 추가`, `docs: 리드미 수정`

### 2. 풀 리퀘스트(PR) 규칙
- 기능을 완성하면 main 브랜치로 바로 푸시하지 말고, 본인의 **작업 브랜치를 파서 Push 후 PR을 생성**해 주세요.
- PR을 생성하면 **자동으로 적용되는 템플릿 양식**이 있습니다. 동기들이 변경 사항을 쉽게 확인할 수 있도록 양식에 맞게 내용을 꼭 작성해 주세요!

### 3. 이슈(Issue) 규칙
- 버그를 발견하거나 새로운 기능 구현을 제안할 때는 깃허브의 **`Issues` 탭에 등록**해 주세요.
- `New Issue` 버튼을 누르면 **`Bug Report`와 `Feature Request` 전용 템플릿**이 자동으로 나타납니다. 양식에 맞춰 작성하면 제목 양식과 라벨이 자동으로 지정되니 편하게 작성해 주시면 됩니다!
